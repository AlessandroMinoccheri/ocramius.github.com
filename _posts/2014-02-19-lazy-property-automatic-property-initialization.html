Yesterday, I worked on yet another interesting little experiment, which is called LazyProperty .

The problem with "lazy" properties

The idea is very simple: avoid manually checking object properties to see if they were initialized.

Let's make a simple example:

{% highlight php %}getUserRepository()->add($user); // ... } // ... protected function getUserRepository() { // we're not using DI here because of performance implication // normally, people use a service locator here for RAD return $this->userRepository ?: $this->userRepository = new MemoryUserRepository(); } }{% endhighlight %}

This is fairly simple API, and it is perfectly fine to use it when we don't want to be forced to build the MemoryUserRepository instance for calls to API methods that don't use it.

There is a problem though, which is that any direct access to UserService#$userRepository has to be avoided: all of the class' implementation as well as its subclasses have to rely on the protected getter in order to access the repository instance.

For example, the following code works only because of a lucky combination of events:

{% highlight php %}getUserRepository()->add($user); } public function login($userId) { $this->userRepository->find($userId); // ... } }{% endhighlight %} {% highlight php %}register($user); $userService->login($user->getId());{% endhighlight %}

This code will run, but only because UserService#register() was called before UserService#login().

Possible solutions

The code in the examples exposes a silly bug, and it should be fixed by avoiding any property access to UserService#$userRepository.

Moreover, implementors of subclasses are also experiencing the issue if they try accessing the protected un-initialized property manually, but the fix for that is to just define UserService#$userRepository as private and being very defensive about its usage.

These are all valid solutions, but we still don't get around using a protected getter in our class' scope.

That's where I had this possibly crazy idea: making the property itself "lazy" and avoiding the getter call completely, making the getter an implementation detail.

Let's see how this is done with the library:

{% highlight php %}initLazyProperties(['userRepository']); } public function register($user) { // ... // now use the property directly $this->userRepository->add($user); // ... } public function login($userId) { $this->userRepository->find($userId); // ... } protected function getUserRepository() { return $this->userRepository ?: $this->userRepository = new MemoryUserRepository(); } }{% endhighlight %}

By calling LazyProperty\LazyPropertiesTrait#initLazyProperties(), we've made sure that any access to the un-initialized UserService#$userRepository will actually trigger a call to UserService#getUserRepository(), and therefore initialize it.

With that, we don't need to actually worry about calling the getter: both the getter call and property access will work the same way, which is pretty cool!

Under the hood

So what is going on? I'm simply exploiting an feature of the PHP language on which I've already blogged at Property Accessors in PHP Userland .

As a reference, here is the annotated body of LazyProperty\LazyPropertiesTrait#initLazyProperties() :

{% highlight php %}lazyPropertyAccessors[$lazyProperty] = false; // if the property is defined, then ignore it (we don't want to alter object state) if (! isset($this->$lazyProperty)) { // unset the property, this allows us to use magic getters unset($this->$lazyProperty); } } }{% endhighlight %}

Quite simple! Nothing really incredible going on here. When is the property actually initialized?

The other method defined by the trait is the Magic getter LazyProperty\LazyPropertiesTrait#__get() :

Conclusions

This "hack" is simple, clean, easily tested and doesn't cause any problems. It works on PHP 5.4, 5.5, 5.6 and even on HHVM, which I wasn't expecting.

Should you use it? Yes, why not?

Do you need it? Usually not: being defensive about usage of your APIs can actually avoid sloppy mistakes like the ones that I've described in the examples.

Clean dependency injection and eventually proxying dependencies produces better results than explicit lazy-loading coded in your classes.

The most interesting thing in this experiment is that PHP yet again allows to implement language-level features in userland, which is awesome!
